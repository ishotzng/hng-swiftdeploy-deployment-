# SwiftDeploy

> Declarative infrastructure manager — the manifest is the single source of truth.

SwiftDeploy reads `manifest.yaml` and generates your entire stack configuration automatically. No handwritten Nginx configs. No manual Docker Compose files. Edit the manifest, run the CLI, done.

## Architecture

```
manifest.yaml
     │
     ▼
swiftdeploy (CLI)
     │
     ├── templates/nginx.conf.tmpl       ──►  nginx.conf
     └── templates/docker-compose.yml.tmpl ──►  docker-compose.yml
                                                       │
                                         ┌─────────────┴──────────────┐
                                         │                            │
                                   [nginx:8080]            [app:3000 (internal)]
                                         │                            │
                                         └──── reverse proxy ─────────┘
```

All external traffic enters through **Nginx on port 8080**. The app container is never exposed directly to the host.

## Prerequisites

Make sure the following are installed on your server:

- Docker (v24+)
- Docker Compose v2 (`docker compose` not `docker-compose`)
- Python 3.8+
- `pyyaml` — auto-installed by the CLI if missing

## Project Structure

```
swiftdeploy/
├── manifest.yaml                    ← Edit ONLY this file
├── swiftdeploy                      ← CLI executable
├── Dockerfile                       ← Builds the app image
├── app/
│   └── main.py                      ← Python HTTP service
├── templates/
│   ├── nginx.conf.tmpl              ← Nginx config template
│   └── docker-compose.yml.tmpl      ← Compose config template
├── nginx.conf                       ← Generated by init (gitignored)
├── docker-compose.yml               ← Generated by init (gitignored)
└── README.md
```

> `nginx.conf` and `docker-compose.yml` are **generated files** — never edit them manually. They are gitignored and regenerated on every `init`.

## Quick Start (AWS EC2)

### 1. Launch an EC2 Instance

- **OS:** Ubuntu Server 24.04 LTS
- **Instance type:** `t2.medium` (minimum)
- **Storage:** 20 GB
- **Security group inbound rules:**

| Type       | Port | Source            |
|------------|------|-------------------|
| SSH        | 22   | Your IP           |
| Custom TCP | 8080 | Anywhere (0.0.0.0/0) |

### 2. Connect via SSH

```bash
chmod 400 ~/Downloads/swiftdeploy-key.pem
ssh -i ~/Downloads/swiftdeploy-key.pem ubuntu@YOUR_EC2_PUBLIC_IP
```

### 3. Install Dependencies

```bash
sudo apt update && sudo apt upgrade -y
sudo apt install -y python3 python3-pip git curl docker.io docker-compose-plugin

sudo systemctl start docker
sudo systemctl enable docker
sudo usermod -aG docker ubuntu

# Log out and back in so the docker group takes effect
exit
```

Reconnect and verify:

```bash
docker --version          # Docker version 24+
docker compose version    # Docker Compose version 2+
python3 --version         # Python 3.8+
```

### 4. Clone the Repository

```bash
git clone https://github.com/YOURUSERNAME/swiftdeploy.git
cd swiftdeploy
```

### 5. Build the Docker Image

```bash
docker build -t swift-deploy-1-node:latest .
```

### 6. Deploy

```bash
./swiftdeploy deploy
```

Your stack is now live at `http://YOUR_EC2_PUBLIC_IP:8080`.

## Manifest Reference

`manifest.yaml` is the **only file you should edit**. Everything else is derived from it.

```yaml
services:
  image: swift-deploy-1-node:latest   # Docker image to run
  port: 3000                          # Internal app port (never exposed directly)
  mode: stable                        # Deployment mode: stable | canary
  version: "1.0.0"                    # Injected as APP_VERSION env var
  restart_policy: unless-stopped      # Docker restart policy
  log_volume: swiftdeploy-logs        # Named volume for logs

nginx:
  image: nginx:latest
  port: 8080                          # Host-facing public port
  proxy_timeout: 30                   # Upstream proxy timeout in seconds

network:
  name: swiftdeploy-net               # Docker network name
  driver_type: bridge                 # Docker network driver

contact: "ops@swiftdeploy.local"      # Used in JSON error response bodies
```

## CLI Subcommands

### `init`

Parses `manifest.yaml` and generates `nginx.conf` and `docker-compose.yml` from templates.

```bash
./swiftdeploy init
```

The grader will delete generated files and re-run this command — everything regenerates correctly from the manifest.

---

### `validate`

Runs **5 pre-flight checks**. Exits with a non-zero code on any failure.

```bash
./swiftdeploy validate
```

| # | Check |
|---|-------|
| 1 | `manifest.yaml` exists and is valid YAML |
| 2 | All required fields are present and non-empty |
| 3 | The Docker image referenced in the manifest exists locally |
| 4 | The Nginx port is not already bound on the host |
| 5 | The generated `nginx.conf` is syntactically valid |

**Example output:**

```
──────────────────────────────────────────────────
 swiftdeploy validate
──────────────────────────────────────────────────

[1/5] manifest.yaml exists and is valid YAML
  ✔ manifest.yaml found and parsed successfully

[2/5] Required fields are present and non-empty
  ✔ services.image = swift-deploy-1-node:latest
  ✔ services.port = 3000
  ✔ nginx.image = nginx:latest
  ✔ nginx.port = 8080
  ✔ network.name = swiftdeploy-net
  ✔ network.driver_type = bridge

[3/5] Docker image exists locally
  ✔ Image found: swift-deploy-1-node:latest

[4/5] Nginx port not already bound
  ✔ Port 8080 is free

  ✔ nginx.conf syntax is valid

```

---

### `deploy`

Runs `init`, brings up the full stack, and blocks until `/healthz` returns `ok` or 60 seconds elapse.

```bash
./swiftdeploy deploy
```

---

### `promote`

Switches the deployment mode between `stable` and `canary` with a rolling restart of the **app container only** (Nginx stays up, no downtime).

```bash
./swiftdeploy promote canary
./swiftdeploy promote stable
```

What it does step by step:

1. Updates `mode` in `manifest.yaml` in-place
2. Regenerates `docker-compose.yml` with the new `MODE` environment variable
3. Restarts the `app` container only
4. Confirms the new mode is active by polling `/healthz`

---

### `teardown`

Stops and removes all containers, networks, and volumes.

```bash
# Remove containers, networks, volumes
./swiftdeploy teardown

# Also delete generated config files
./swiftdeploy teardown --clean
```

## API Endpoints

| Method | Path      | Description                                      | Mode         |
|--------|-----------|--------------------------------------------------|--------------|
| GET    | `/`       | Welcome message with mode, version, timestamp    | All          |
| GET    | `/healthz`| Liveness check — returns status, mode, uptime   | All          |
| POST   | `/chaos`  | Simulate degraded behaviour                      | Canary only  |

### `GET /`

```json
{
  "message": "Welcome! Running in stable mode",
  "mode": "stable",
  "version": "1.0.0",
  "timestamp": "2026-05-05T10:30:00Z"
}
```

### `GET /healthz`

```json
{
  "status": "ok",
  "mode": "stable",
  "uptime_seconds": 142.5
}
```

> In **canary mode**, every response includes an `X-Mode: canary` header.

## Chaos Engineering

The `/chaos` endpoint is only active in **canary mode**. It accepts a JSON body to simulate degraded behaviour.

### Slow mode — delay responses by N seconds

```bash
curl -X POST http://localhost:8080/chaos \
  -H "Content-Type: application/json" \
  -d '{"mode": "slow", "duration": 3}'
```

### Error mode — return HTTP 500 on ~50% of requests

```bash
curl -X POST http://localhost:8080/chaos \
  -H "Content-Type: application/json" \
  -d '{"mode": "error", "rate": 0.5}'
```

### Recover — cancel all active chaos

```bash
curl -X POST http://localhost:8080/chaos \
  -H "Content-Type: application/json" \
  -d '{"mode": "recover"}'
```

## Nginx Access Log Format

```
$time_iso8601 | $status | ${request_time}s | $upstream_addr | $request
```

**Example:**

```
2026-05-05T10:30:01+00:00 | 200 | 0.002s | 172.18.0.3:3000 | GET / HTTP/1.1
2026-05-05T10:30:05+00:00 | 200 | 0.001s | 172.18.0.3:3000 | GET /healthz HTTP/1.1
```

View live logs:

```bash
docker logs $(docker ps --filter "name=nginx" -q)
```

## Security

| Measure | Detail |
|---------|--------|
| Non-root user | App runs as `appuser` inside the container |
| Dropped capabilities | `cap_drop: ALL` on both containers |
| Minimal re-add | Only `NET_BIND_SERVICE` re-added to Nginx |
| No direct port exposure | App port `3000` is never bound to the host |
| Lightweight image | Python 3.12 Alpine base — well under 300 MB |
| Named volumes only | No bind-mounting sensitive host paths |
